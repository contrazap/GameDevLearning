## Mental model

**Epic ships Linux UE as a zip, not a launcher.** There is no Epic Games Launcher on Linux. The [Unreal Engine for Linux page](https://www.unrealengine.com/en-US/linux) (Epic login required) serves a prebuilt archive per version — 5.8.0: 37.08 GiB compressed, 73 GiB / 296,565 files unpacked (older docs said ~25/43 GB). Unzip anywhere user-writable, run `Engine/Binaries/Linux/UnrealEditor`, done. No install step, no registry: "installed" means "extracted". The engine writes caches (DDC, shaders) into its own tree and `~/.config/Epic/`, which is why it lives in your home dir, not `/opt`. The prebuilt binaries are built against an old glibc baseline (Rocky/RHEL 8), so they run forward-compatibly on Arch's newer glibc.

## Steps

1. ✅ **[You] Packages.**
   ```bash
   sudo pacman -S --needed git-lfs unzip uv
   git lfs install     # machine-wide, once — installs the LFS filters S0.1's .gitattributes rules need
   ```
   **Done when:** `git lfs env` runs clean; in `A1_Flappy`, `git lfs ls-files` returns empty *without error* (empty is correct — no binary assets committed yet); `uvx --version` prints; **and `A1_Flappy/.git/hooks/pre-push` exists**.
   **Status 2026-07-23 — ✅ (trap caught, then fixed):** packages in (git-lfs 3.7.1, unzip 6.0, uv 0.11.31), global filters set, `env`/`ls-files` clean — but `A1_Flappy` initially had **no LFS hooks**: the clone predates git-lfs on this machine, and machine-wide `git lfs install` doesn't retrofit existing repos. Fixed by running `git lfs install` *inside* the repo; verified: all four LFS hooks present, `pre-push` calls `git lfs pre-push`, endpoint resolves to `github.com/contrazap/A1_Flappy`. Why it matters: filters alone make commits into pointers, but only the `pre-push` hook uploads the binary objects — without it, pushes silently strand GitHub with pointers to nothing, breaking clone-=-resume. Applies to every future project cloned onto a machine before its `git lfs install`.
