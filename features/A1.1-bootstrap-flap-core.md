## Why & what

**Goal:** a bird that flies and dies believably — placeholder pawn under gravity, one-button flap through Enhanced Input, run ends on ground/ceiling/pipe contact — with gravity and impulse as tunable editable defaults (A1 project DoD).

**Design rationale (why Flappy Bird is built this way):** Flappy Bird is one tuning relationship wearing a game as a costume — the gravity/impulse ratio *is* the product. One button, one axis of motion: the bird never moves forward; the world will move past it (A1.2), which keeps the camera fixed and the math one-dimensional. Death-on-any-touch means collision rules stay binary — no health, no partial hits — so the difficulty lives entirely in level geometry and tuning values. That's why this feature obsesses over *editable defaults* rather than more mechanics: in this genre, tuning access is the game design. Per locked decision 6 it's camera-constrained 3D, not Paper2D — the 3D scene/actor/collision reps here are the same ones every later ladder project uses.

**Scope — In:** our own map (startup + game default), fixed side-view camera, `BP_Bird` pawn with placeholder engine mesh, GameMode wiring, gravity + flap via Enhanced Input, death on ground/ceiling/static-test-pipe contact, tunable defaults.
**Out:** pipe spawner/movement, scoring, restart flow, audio (all A1.2); UI/save (A1.3); any real art.
**Cut line:** believable death handling (tumble/feedback) can shrink to "input stops, bird drops"; tunable gravity/impulse cannot be cut.

**Assets needed:** none external — placeholder mesh from engine basic shapes (`/Engine/BasicShapes`); everything else is BP + level geometry.

## Mental model

1. **The possession chain.** A map declares (or inherits) a **GameMode**; the GameMode names a **default pawn**; at play start a **PlayerController** spawns and *possesses* that pawn. Input, camera, and control all hang off this chain — when something "doesn't respond", walk the chain first. Our chain: `FlappyMain` map → `BP_FlappyGameMode` → `BP_Bird`.
2. **Enhanced Input is a routing pipeline, not key bindings.** Hardware keys map to an **Input Action** (a semantic event: "Flap") inside an **Input Mapping Context** (a swappable binding set); the context is *added at runtime* to the local player's Enhanced Input subsystem, and the pawn listens for the action event. The indirection buys rebinding, contexts-per-mode (menu vs play), and multi-device mapping — one context is trivial now; the shape is what you're learning. Already the project default: [DefaultInput.ini:79-80](../../GameDevLearningPractice/A1_Flappy/Config/DefaultInput.ini) points at `EnhancedPlayerInput`/`EnhancedInputComponent` (verified 2026-07-22).
3. **Two ways to fall.** (a) *Physics simulation:* the body ragdolls under world gravity, flap = `Add Impulse`; free believability, but tuning is indirect — 5.8 rigid bodies have **no per-body gravity scale** (only on/off + gravity groups; verified in engine source 2026-07-22), so "heavier" means world-gravity edits or added force. (b) *Hand-integrated velocity:* a float `VelocityZ` you update each Tick (`v += g·Δt`, offset actor by `v·Δt`), flap = set `v = FlapImpulse`; two directly-editable numbers, deterministic feel, but collision response is yours to handle. This is Session 2's design decision — Flappy's feel lives or dies here. Units: 1 UU = 1 cm; engine world gravity is −980 UU/s².
4. **Hit vs overlap.** Blocking collision + **hit events** = "I struck a wall" (needs *Simulation Generates Hit Events* when physics-driven); **overlap events** = "I passed through a trigger". Death-on-contact wants blocking hits; A1.2's score gates will want overlaps — you're setting up both mental slots now.
5. **The stage, not a world.** One static camera looking down the Y axis at an X-Z playfield; the bird's X never changes. Ground and ceiling are just blocking geometry placed once. Flappy's scrolling is an illusion the *pipes* perform (A1.2) — nothing about the camera or bird ever moves horizontally.

### Teach

**1 — A map is an asset; "which map" is two settings, not one.** A `.umap` stores placed actors plus level settings. The project decides maps twice: **Editor Startup Map** (what the editor opens) and **Game Default Map** (what a packaged/PIE-standalone game boots into). Both live in Project Settings → Maps & Modes and are written to `Config/DefaultEngine.ini` under `[/Script/EngineSettings.GameMapsSettings]`. Our Blank template shipped with `GameDefaultMap` pointing at the engine's OpenWorld template and no startup map at all — today both become `FlappyMain`.

**2 — The possession chain, concretely.** Press Play and this fires in order: the map asks *which GameMode?* (per-map World Settings override, else the project's **Default GameMode**) → the GameMode spawns its **Default Pawn Class** at a **PlayerStart** actor → a PlayerController is created and **possesses** that pawn → the controller picks a view. Our chain: `FlappyMain` → `BP_FlappyGameMode` → `BP_Bird` at the PlayerStart → possessed → view = our camera (below). Every "nothing happens when I press Play" bug in this project will be a break in one of these links — always debug by walking the chain in this order.

**3 — Pawn anatomy: the root component matters.** A Pawn is a components tree; the **root component** defines the actor's transform, and *physics/impulses apply to the root*. A fresh Pawn BP has a placeholder `DefaultSceneRoot` (pure transform, no visual, no collision). We make the bird's Static Mesh the root — in Session 2, gravity/impulse (either movement option) must act on the mesh's body, and a mesh buried under a scene root would silently not move the actor.

**4 — One static camera = one checkbox.** A PlayerController renders whatever its **view target** is — by default the possessed pawn. A placed **CameraActor** with *Auto Activate for Player = Player 0* claims the view target at startup with zero Blueprint (verified in 5.8 source: `CameraActor.h:24`). The BP alternative (`Set View Target with Blend` on BeginPlay) buys runtime camera switching we will never need — Flappy's camera never moves. Camera default FOV is 90° horizontal: at 1200 UU from the playfield plane the window is ~2400 UU wide — that's the number sizing our stage.

**5 — Engine content as placeholder art.** `/Engine/BasicShapes/` (Cube, Sphere, Cone…, verified on disk) ships with the engine; show it via the Content Browser → Settings (gear) → **Show Engine Content**. Engine assets are *referenced*, not copied into `Content/` — fine for placeholders since every machine with UE 5.8 has them; real imported assets follow import-lean + `ASSETS.md` instead (CONVENTIONS rule 5). Naming note: `BP_` prefix per CONVENTIONS §Blender naming table.

**6 — Stage coordinates (used by every later session).** X = horizontal travel axis (bird's X never changes — Mental model §5), Y = camera depth axis, Z = up; 1 UU = 1 cm. Today's numbers: floor top at Z=0 (template floor), ceiling underside ~Z=725, bird spawn (0, 0, 350), camera (0, −1200, 350) with yaw 90° so it faces +Y at the playfield plane Y=0.

## Session 2 — Gravity & flap (Enhanced Input)

The feel session. Opens with the feature's one genuine **design decision** ([You], logged): physics-simulated body vs hand-integrated velocity — trade-offs in Mental model §3, decided *before* anything is built on it. Then the Enhanced Input first encounter at guided grain: `IA_Flap` + `IMC_Flappy` assets, context added to the subsystem on the pawn's BeginPlay, action event driving the flap; gravity + impulse exposed as editable, instance-tunable defaults on `BP_Bird` (project DoD: no magic numbers in graphs). Constrain motion so the bird cannot leave the X-Z plane or rotate off-axis (mechanism depends on the design decision). Ends with a tuning pass — flap-to-fall ratio until it *feels* like Flappy. **End state:** PIE — bird falls, Space (and left mouse) flaps, values tunable on the instance without opening the graph.

**Teach seed** (*re-verify at session start*): project input classes already Enhanced Input (verified — Mental model §2); the runtime add is `Get Enhanced Input Local Player Subsystem → Add Mapping Context` from the pawn's controller; world gravity default −980 UU/s² (`DefaultGravityZ`, engine `BaseEngine.ini`), editable per-project under Project Settings → Physics if the physics-sim route is chosen.

## Session 3 — Dying believably

Collision ends the run. Ground/ceiling contact kills; add one **static test pipe** (placeholder mesh, hand-placed — the spawner is A1.2's) so pipe-death is proven before pipes exist in number. First-encounter skills: collision presets/object responses, hit events (and *Simulation Generates Hit Events* if physics-driven), a minimal `bIsDead` gate — dead birds ignore input and stop flapping, with a believable death read (drop/tumble — cut-line: a plain drop suffices). No restart flow: that's A1.2's game-state work; PIE restart is the loop for now. Ends with the feature's full tuning-and-verification pass against the DoD, then feature close ritual (KN conversation, ROADMAP bump, commit + push both repos [You]). **End state:** feature DoD passes — flies and dies believably, ground/ceiling/pipe all lethal, tunables editable.
