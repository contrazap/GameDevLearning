# F1 — Capstone

**Folder/repo:** `F1_<Title>` (named at F1.1) · **Clone of:** your pick of the 18 · **Platform:** Linux (+📱 if the title fits) · **Est:** 20 sessions · **Everything:** story mode

**Why this project:** the graduation exam. One reference game, cloned properly as a vertical slice — the first 10 minutes that *feel like that game*, shipped. **Every feature runs in story mode**: requirements and acceptance criteria only, no steps, no Teach blocks except a genuine first-encounter island if the title demands one (it shouldn't — pick a title the ladder covered). Claude is consultant (escalation ladder) and reviewer, nothing more. Passing this = the roadmap's goal achieved: you can take up any of the clones on your own.

**Choosing the title (at F1.1, not before):** pick for *coverage overlap*, not novelty — the best capstone reuses 5+ threads at rep 4+. Candidates by then-covered threads: a fuller Souls arena, a GTFO-solo rundown, a Zomboid slice, a SnowRunner map, a polished Vampire Survivors. A title needing a thread the ladder never opened (true procgen, networked anything) is out of scope by definition.

**Scope — In:** the slice spec you write at F1.1 (with cut lines), full game loop, front end (menus/settings/save), performance pass against stated budgets, packaged and shippable to a stranger.
**Out:** content breadth beyond the slice, live-ops anything, store submission (post-roadmap).
**Cut line:** written by you at F1.1 — writing honest cut lines is part of the exam.

## Features

### F1.1 — Concept & spec (2)
Pick the title; write the slice spec *as a story-mode plan for yourself*: the 10-minute experience, feature list with acceptance criteria, asset plan (free/Blender/reuse), budget targets, cut lines. Claude reviews the spec like a producer would: scope honesty, criteria testability, risk flags. The approved spec becomes F1.2–F1.5's requirements.

### F1.2 — Core loop slice (6)
The mechanics that make it *that game*, playable start to end at greybox fidelity. Acceptance criteria from your spec.

### F1.3 — Content & systems pass (6)
The data, enemies/levels/goods, and system depth that make the loop repeatable for 10 honest minutes.

### F1.4 — Front end, save & settings (3)
Menus, options (audio/video/controls), persistence — the unglamorous 20% that separates a project from a game.

### F1.5 — Perf, polish & ship (3)
Profiling pass against your stated budgets, the juice pass, packaged build(s), and a README for the repo — written for a stranger who downloads it.

## Project DoD additions
- A stranger plays the packaged build for 10 minutes and names the reference game unprompted.
- Every F1 feature closed without escalation past level ② (concept) — the honest measure of independence.
- Post-mortem KN note: what the ladder taught well, what it didn't — the input to whatever comes next (multiplayer roadmap).
